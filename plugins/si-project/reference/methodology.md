# SI 프로젝트 산출물 방법론 지식

> 본 문서는 **워크플로우 강제용이 아닌 출력 형식 보장용**이다. 개발은 AI 에이전트와 자유롭게 진행하고, 발주처 요구 시점에 본 가이드를 기반으로 산출물을 생성한다.

---

## 0. 명확성 등급 (Clarity Grade) — 모든 요구사항·산출물 항목 표기 의무

모든 요구사항·산출물의 개별 항목(기능·비기능·인터페이스·데이터·NFR 등)은 다음 3등급으로 명확도를 표기한다. AI 에이전트가 추측 구현으로 빠지는 것을 막기 위한 게이트.

| 등급 | 의미 | 구현 가능 여부 |
|------|------|------------|
| **명확** | 구현·테스트에 필요한 정보가 충분. 추측 불필요 | 가능 |
| **모호** | 일부 정보 누락. 추측 필요한 영역 존재 | **금지** — 명확화 또는 사용자 확인 후 |
| **미정** | 항목 자체가 결정되지 않음 | **금지** — 결정 후 |

산출물 표 형식 예시:

| ID | 제목 | 설명 | 우선순위 | 명확도 | 출처/근거 |
|----|------|------|---------|--------|----------|
| FR-001 | 사용자 가입 | 이메일·비밀번호로 가입 | 상 | 명확 | RFP §3.1 |
| FR-002 | 상품 추천 | 사용자에게 상품 추천 | 중 | 모호 | 추천 알고리즘 미정 |
| FR-003 | 다국어 지원 | 한/영/일 | 하 | 미정 | 발주처 협의 중 |

AI 에이전트 작업 시:
- **`명확` 항목만 구현 진행**
- `모호`/`미정` 항목은 `## 미설정 항목` (CLAUDE.md) 또는 `docs/06-management/change-request.md`에 기록하고 사용자 확인 후 진행
- 구현 직전 `/si-project:project-check`로 등급 카운트·준비도 점수 확인
- 본 등급이 누락된 산출물은 `/si-project:project-document`·`/si-project:project-milestone` 생성 시 자동으로 `명확도` 컬럼 추가

`/si-project:project-check`가 본 등급을 스캔해 준비도 점수를 산출한다 (`명확` 비율 + `.env` 채움률 + 미설정 항목 카운트).

## 1. 방법론 개요

본 방법론은 한국 SI 발주처에서 일반적으로 요구하는 산출물 표준을 기반으로, AI 에이전트 기반 개발 환경에 맞게 정리한 것이다. 단계 게이트 강제가 아닌 **출력 형식 보장**에 초점을 둔다.

### 단계 (8단계로 재정렬)

| # | 단계 | 영문 디렉토리 | 주요 활동 |
|---|------|------------|----------|
| 0 | 착수·계획 | 00-kickoff | 프로젝트 정의·계획·위험 식별 |
| 1 | 요구분석 | 01-requirements | 비전·요구사항·행위자·프로세스 |
| 2 | 아키텍처 | 02-architecture | SW/시스템 아키텍처·보안·데이터 모형 |
| 3 | 상세설계 | 03-design | 컴포넌트·DB·UI·API 설계 |
| 4 | 구현 | 04-implementation | 코딩·단위 테스트·통합 |
| 5 | 시험 | 05-testing | 통합·시스템·인수 테스트 |
| 6 | 관리 | 06-management | 품질·형상·변경·진척 관리(전 단계 횡단) |
| 7 | 인도 | 07-delivery | 설치·전환·교육·운영 인계·종료 |

### 6개 관점 (Six Perspectives)

발주처가 요구하는 다각도 검토를 위한 관점들. 매 문서 생성 시 6관점을 점검한다.

| 관점 | 영문 | 점검 포커스 |
|------|------|------------|
| PMO | PMO | 일정·범위·예산·이해관계자 |
| PM | Project Manager | 인력·리스크·의사소통·이슈 |
| 아키텍처 | Architect | 기술 선택·구조·NFR·확장성 |
| DBA | DBA | 데이터 모델·성능·무결성·백업 |
| 하네스 | Harness Engineer | 빌드·배포·CI/CD·환경 분리 |
| 보안 | Security Specialist | 위협 모델·인증·암호화·감사 |

> AI 에이전트 개발 환경에서는 1~2인이 다역할을 수행. 관점은 검토 체크리스트로 활용.

## 2. 단계별 산출물 매트릭스

각 단계 진입 시 발주처가 일반적으로 요구하는 필수 산출물. 상세 100개 목록은 `doc-catalog.md` 참조.

### 00-kickoff (착수 시)
필수: 프로젝트정의서·프로젝트수행계획서·위험분석서·프로젝트개발환경정의서

### 01-requirements (요구분석 완료 시)
필수: 요구사항기술서·시스템비전기술서·사용자정의서·용어집

### 02-architecture (아키텍처 완료 시)
필수: 소프트웨어아키텍처기술서·시스템아키텍처정의서·보안정의서·데이터모형기술서

### 03-design (상세설계 완료 시)
필수: 컴포넌트명세서·데이터베이스설계서·테이블정의서·화면정의서

### 04-implementation (구현 완료 시)
필수: 제품 백로그·완료 기준(DoD)

### 05-testing (테스트 완료 시)
필수: 테스트계획서·테스트설계서·테스트수행보고서·테스트결과보고서

### 06-management (지속적)
필수: 변경요청·품질관리계획서·형상관리절차서·프로젝트진척보고서

### 07-delivery (인도 시)
필수: 시스템설치보고서·데이터전환보고서·시스템전환계획서·사용자지침서·운영계획서·프로젝트완료보고서

## 3. 핵심 문서 작성 가이드

`project-document` 스킬이 각 문서 생성 시 본 절을 참조해 섹션 구조·표 컬럼·다이어그램을 결정한다.

> **Lazy-load 패턴 (v2.4)**: 전체 섹션을 읽지 말 것. 필요한 문서의 마커만 Grep으로 위치 잡고 그 범위만 Read.
>
> ```
> # 예: project-document 스킬이 db-design.md 생성 시
> Grep pattern="<!-- DOC: db-design -->"   → 시작 라인 N 확보
> Grep pattern="<!-- /DOC: db-design -->"  → 종료 라인 M 확보
> Read offset=N, limit=(M-N+1)             → 해당 섹션만 로드 (~20-40줄)
> ```
>
> 카탈로그 외 문서는 일반 작성 원칙(섹션 4 이후)을 참조한다.

- **애자일 통합**: 제품백로그·DoD·스프린트템플릿을 04-implementation에 추가

## 5. 납품 형식 규약

### 5.1 마크다운 → 다른 포맷 변환

발주처 요구 시 다음 변환 옵션:
- `pandoc input.md -o output.pdf` (PDF 변환)
- `pandoc input.md -o output.docx` (Word 변환)
- Mermaid 다이어그램은 변환 전 PNG로 미리 렌더링 권장

### 5.2 문서 헤더 표준

모든 산출물은 YAML 프론트매터로 메타 포함:
```yaml
---
title: 한글 정식 명칭
original-kr: 원본 한글 명칭
project: "{{프로젝트명}}"
milestone: 01-requirements
version: 1.0
status: 완료 / 검토중 / 초안
date: "YYYY-MM-DD"
owner: "담당자명"
---
```

### 5.3 상태 흐름

`초안` → `검토중` → `검토완료` → `승인완료` → `완료`

각 단계 전환 시 변경 요청 절차 필요.

### 5.4 인쇄 시 주의사항

- 한글 폰트 임베드 (`맑은 고딕` 또는 `나눔고딕`)
- 표는 페이지 분할되지 않게 크기 조정
- Mermaid 다이어그램은 SVG 변환 후 삽입
